项目概述

本项目是一个面向【校园师生】的【物品捐赠管理系统】，主要解决【校园闲置物品流转效率低的问题】。系统采用【前后端分离 B/S 架构】，支持用户注册（管理员审核）、物品发布与 6 种状态流转、捐赠申请与交易交付、互评信用等完整业务流程。

我在项目中负责【系统架构设计、数据库建模（13 张表）、后端核心业务逻辑实现及性能优化】等工作，重点完成了【用户权限与鉴权模块】、【物品管理与状态流转模块】以及【基于协同过滤的推荐算法】。

系统架构

系统整体采用前后端分离的 B/S 架构，后端遵循 Controller → Service → Mapper 三层分层设计。前端通过 Axios 调用 RESTful API，使用 JWT 进行无状态鉴权，Pinia 管理前端状态，路由守卫实现权限隔离。

基本流程：用户通过浏览器访问 Vue3 前端页面 → 前端调用后端 RESTful API（携带 JWT Token）→ Controller 层接收请求并校验参数 → Service 层处理业务逻辑 → Mapper 层与 MySQL 交互 → Redis 缓存热点数据减少数据库压力 → 系统返回处理结果并在前端渲染展示。

核心业务流程

物品从发布到完成捐赠经历完整的生命周期管理：

用户上传物品信息（多图）→ 系统自动进入待审核状态

2

3

5
互评信用

交易双方互相评分（1–10 分）→ 信用资料卡更新 → Redis 缓存

技术难点与解决方案

难点：物品列表多表关联查询响应慢

问题描述：在开发过程中，物品列表页需要关联 sys_user、donation_item、category、donation_request 等多表进行查询，随着数据量增长，页面加载时间超过 3 秒。

解决方案：

  • 开启 MySQL 慢查询日志，捕获瓶颈查询
  • 使用 EXPLAIN 分析执行计划，定位到多表关联查询未走索引
  • 为高频查询字段（物品状态、分类 ID、发布时间）添加联合索引
  • 引入分页查询机制，控制单次返回数据量
  • 对推荐结果等热点数据使用 Redis 缓存，减少 MySQL 查询压力

最终效果：优化后列表查询时间由 3.2 秒降至 0.4 秒，查询效率提升约 30%，Redis 缓存大幅减少 MySQL 压力。

优化成果

项目在完成完整功能的基础上，进行了系统性的性能优化。

量化成果：

  • 联合索引 + 分页查询 → 列表查询 3.2s → 0.4s（提升约 30%）
  • Redis 缓存热点数据 → 大幅减少 MySQL 压力
  • WebSocket 实时通知 → 通知推送即时性提升
  • 设计 13 张数据库表 → 覆盖全业务流程
  • 3 种用户角色 → 权限分级隔离管控
  • 4 种交易状态 → 完整追踪捐赠交付链路
